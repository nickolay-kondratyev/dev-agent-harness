## What Changed

Restored the staleness guard for `context_window_slim.json` that was previously removed as "benign".

### Spec changes (doc/use-case/ContextWindowSelfCompactionUseCase.md)
- Updated vocabulary: `context_window_slim.json` format now documents `file_updated_timestamp` field
- `ContextWindowState.remainingPercentage` is now `Int?` — null = stale/unknown
- `ClaudeCodeContextWindowStateReader`: when `file_updated_timestamp` is older than `contextFileStaleTimeout`, return `remainingPercentage = null` + log warning
- Both compaction trigger sites must skip compaction on null (log warning instead)
- Restored `contextFileStaleTimeout` row in Thresholds table (5 min default)
- Updated Risks section to reflect detection via timestamp

### Code changes (HarnessTimeoutConfig.kt)
- Added `contextFileStaleTimeout: Duration = 5.minutes` field
- Added fast-test override `contextFileStaleTimeout = 2.seconds` in `forTests()`

## Why

A stale hook writing a frozen `remaining_percentage` that looks healthy would prevent compaction from triggering, causing the agent to silently run out of context. This is NOT benign — it causes crashes, triggers recovery, and wastes time. The `file_updated_timestamp` field in the JSON provides a reliable signal for staleness detection without relying on OS file mtime.
