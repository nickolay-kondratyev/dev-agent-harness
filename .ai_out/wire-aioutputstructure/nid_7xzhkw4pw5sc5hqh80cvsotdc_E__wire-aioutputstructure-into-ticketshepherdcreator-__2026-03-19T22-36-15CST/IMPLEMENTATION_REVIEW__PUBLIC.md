# Implementation Review: Wire AiOutputStructure into TicketShepherdCreator

## Summary

Single commit (`e11e019`) wires `AiOutputStructure` into three locations:

1. **`TicketShepherdCreatorImpl`** -- receives `AiOutputStructure` + `WorkflowDefinition`, calls `ensureStructure()` during `create()`.
2. **`ContextForAgentProviderImpl`** -- receives `AiOutputStructure` as a constructor param (unused for now, wired for future path resolution).
3. **`TicketShepherdDeps`** -- carries `AiOutputStructure` so `TicketShepherd` collaborators can access it.

Also adds `allPartsForStructure()` to `WorkflowDefinition` to provide the canonical parts list for directory creation.

**Overall assessment**: Clean, well-tested implementation. The wiring is correct, tests are comprehensive, and all tests pass. Two issues worth addressing below.

## IMPORTANT Issues

### 1. KDoc claims "Pure wiring -- no side effects" but `create()` now performs I/O

**File**: `/home/nickolaykondratyev/git_repos/nickolay-kondratyev_dev-agent-harness-mirror-1/app/src/main/kotlin/com/glassthought/shepherd/core/TicketShepherdCreator.kt`

The `TicketShepherdCreator.create()` KDoc on the interface (line 60) says:

```kotlin
/**
 * Wires ticket-scoped dependencies. Pure wiring -- no side effects.
 */
fun create(): TicketShepherdCreatorResult
```

And the class-level KDoc (line 41) says:

```
* Pure wiring only -- no side effects.
```

But the implementation now calls `aiOutputStructure.ensureStructure(...)` which creates directories on disk via `Files.createDirectories()`. This is a filesystem side effect. The documentation should be updated to reflect the new reality -- something like "Wires ticket-scoped dependencies and sets up the `.ai_out/` directory structure."

This matters because a caller reading the interface KDoc would reasonably assume `create()` is safe to call without filesystem consequences, which is no longer true.

## Suggestions

### 1. `allPartsForStructure()` for with-planning workflows returns only planning parts

The method's KDoc and implementation are consistent -- for with-planning workflows it returns `planningParts`. This is correct for the initial directory structure setup because execution parts are dynamic (generated by the planner). However, the method name `allPartsForStructure` could be read as "all parts that will ever need structure." The KDoc is clear enough, but worth noting that when execution parts are resolved later (from `plan_flow.json`), `ensureStructure()` will need to be called again with those parts. This is presumably handled elsewhere but is worth verifying during integration.

### 2. Test for `ensureStructure` being called with correct parts

The `TicketShepherdCreatorTest` (lines 153-211) tests that `ensureStructure` creates the expected directories, which implicitly verifies the correct parts were passed. This is a solid integration-style test. An alternative would be to use a spy/fake `AiOutputStructure` to verify the exact `List<Part>` passed to `ensureStructure()`, but the current approach of verifying the end state (directories exist) is pragmatic and sufficient.

- Update KDoc on `TicketShepherdCreatorImpl` class to remove "Pure wiring only -- no side effects" (line 41).
